# QA 및 테스트 운영 문서

버전: v1.1  
작성일: 2026-03-06  

## 1. 테스트 원칙
- 보안 검증(권한, RLS, 세션)이 기능 검증보다 우선이다.
- 실패 경로(권한 실패/검증 실패/네트워크 실패) 우선 검증한다.
- 테스트는 문서 간 트레이스(요구사항-기능-테스트)로 운영한다.

## 2. 테스트 범위
- 인증 플로우: 회원가입, 로그인, 로그아웃, 재설정
- 레시피 CRUD
- 공개 범위: 공개 상세, 공개 목록, 공개 검색
- 탈퇴 처리: 즉시 삭제
- 이미지 업로드/삭제/재시도
- RLS/권한 우회 시도

## 3. 필수 통과군(Go/No-Go)
- TC-A1~TC-A5, TC-C1~TC-C5, TC-S1~TC-S6, TC-U1~TC-U6, TC-F1~TC-F3
- API 방식(`Route Handler`)과 호출 규격이 문서와 일치해야 함
- 비회원 공개 범위가 `api-spec.md`와 동일해야 함

## 4. 정합성 트레이스
- 요구사항: `requirements.md`
- 기능: `features.md`
- 시나리오: `user-scenarios.md`
- 테스트: `test-cases.md`
- API: `api-spec.md`
- 정합성: `logic-consistency-review.md`

## 5. 결함 심각도
- Critical: 타인 데이터 노출, RLS 미동작, 탈퇴 미삭제
- High: 공개 정책 위반(비로그인 접근 가능), 토큰 노출
- Medium: 저장 실패, 삭제 경합 미처리, 이미지 업로드 연속 실패
- Low: 문구/디자인 미세 오류

## 6. 릴리즈 판단 기준
- Critical/High 0건
- Medium는 대응 계획 명시 또는 수정 완료 후 통과
- Low는 릴리즈 노트로 처리 가능
